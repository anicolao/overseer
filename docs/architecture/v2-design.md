# V2 Architecture & Requirements: Event-Driven Orchestration

## 1. Executive Summary
The V2 evolution of the Overseer project transitions the system from a synchronous, stateless Minimum Viable Product (MVP) to an asynchronous, event-driven orchestration system. This upgrade focuses on enabling complex, multi-step agent workflows, cross-task memory retention, and secure code execution capabilities necessary for production environments.

## 2. User Requirements
* **UR-1: Asynchronous Task Orchestration:** The system must support asynchronous workflows where tasks can be paused, resumed, and routed between specialized agents without blocking the main orchestrator.
* **UR-2: Persistent Context & Memory:** Agents must be able to recall past interactions, project context, and prior code iterations. The system requires both short-term (session) and long-term (semantic) memory.
* **UR-3: Secure Execution Environment:** Any code generated by the agent swarm must be evaluated, tested, and executed in an isolated sandbox to prevent malicious operations or host system compromise.
* **UR-4: Fault Tolerance & Retries:** The architecture must gracefully handle API rate limits, LLM timeouts, and worker crashes via automated retries and dead-letter queues.

## 3. High-Level Technical Design

### 3.1. Architectural Style
The system will adopt an **Event-Driven Architecture (EDA)**. Agents will no longer call each other directly via synchronous HTTP requests. Instead, they will act as isolated workers subscribing to specific topics on a central message broker.

### 3.2. Core Components
1. **The Event Broker (Message Bus):**
   * *Technology:* RabbitMQ or Redis Streams.
   * *Responsibility:* Decouple agents. Route events (e.g., `Task.Created`, `Code.Committed`, `Tests.Failed`) to the appropriate agent worker queues.
2. **The Overseer Orchestrator (State Machine):**
   * *Technology:* Temporal.io or custom PostgreSQL-backed state machine.
   * *Responsibility:* Track the overarching lifecycle of an issue. Ensure SLAs, handle routing logic, and manage global state.
3. **Persistent Memory Layer:**
   * *State & History:* PostgreSQL (stores task metadata, agent comments, and PR statuses).
   * *Semantic Memory:* Vector Database (e.g., Pinecone or Qdrant) to store embeddings of the codebase, documentation, and past architectural decisions for Retrieval-Augmented Generation (RAG).
4. **Secure Execution Sandbox:**
   * *Technology:* Docker-in-Docker (DinD) with strict resource limits, or WebAssembly (WASM/Wasmtime) containers.
   * *Responsibility:* Provide a highly restricted, ephemeral environment where the Developer and Tester agents can compile code, run test suites, and execute scripts without host-level access. Network access will be disabled or strictly allow-listed.
5. **Agent Swarm (Workers):**
   * *Stateless Consumers:* Architect, Planner, Developer, Tester, and Quality agents will run as independent containerized worker processes that consume specific task payloads, interface with external LLMs, and publish results back to the broker.

### 3.3. Component Interaction Flow (Example: Issue Remediation)
1. **Trigger:** A new GitHub issue is labeled `bug`. A webhook triggers the Orchestrator.
2. **Planning:** Orchestrator publishes `Issue.Triage`. The Planner agent consumes it, breaks it into steps, and publishes `Plan.Created`.
3. **Development:** Orchestrator routes the first step to the Developer agent (`Task.Develop`). The Developer pulls codebase context from the Vector DB, writes code, and publishes `Code.Proposed`.
4. **Testing & Sandboxing:** The Tester agent consumes `Code.Proposed`, spins up an ephemeral Docker Sandbox, runs the test suite, and publishes `Tests.Passed` or `Tests.Failed`.
5. **Quality Review:** The Quality agent reviews the diff and test results.
6. **Resolution:** Orchestrator aggregates approvals and commands a Git integration service to merge the PR.

## 4. Migration Strategy & Next Steps
1. Implement the Event Broker and refactor the current synchronous MVP agents into independent worker processes.
2. Integrate the PostgreSQL state database to track issue lifecycle.
3. Establish the Docker-based secure sandbox for the testing agent.
4. Implement the Vector DB memory layer for code context.
