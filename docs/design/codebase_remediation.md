# Codebase Remediation Plan

## Objective
Provide a detailed critique and remediation plan for cleaning up the codebase prototype, focusing on removing extraneous functionality, eliminating duplicate code, and addressing missing test coverage while retaining the path for future cloud deployment.

## Critique

### Future Cloud Deployment Readiness
- **`src/index.ts`**: This file contains the `overseerWebhook` logic using Express. Although the project actively relies on GitHub Actions (`.github/workflows/overseer.yml`) and `src/dispatch.ts` for its execution layer, `src/index.ts` must be retained. Keeping this webhook intact is critical for future deployment to Google Cloud Functions.

### Extraneous Functionality
- **`src/wire_test.ts`**: A stale, standalone script intended to verify Nix VM environments (`ls`, `node --version`, `gh --version`). It is not integrated into `package.json` scripts or CI workflows.

### Missing Test Coverage
Critical system interaction boundaries lack unit tests, leaving the core infrastructure vulnerable to regressions:
- **`src/utils/gemini.ts`**: The Gemini API client layer lacks test coverage.
- **`src/utils/github.ts`**: The GitHub API client wrapper is untested.
- **`src/utils/prompt_files.ts`**: The prompt loading and rendering utility is untested.
- **`src/utils/trace.ts`**: The async local storage tracing setup lacks validation tests.
- **`src/scripts/run_task_persona.ts`**: The local CLI test execution runner is completely untested.

## Remediation Plan

### Increment 1: Remove Stale Scripts
- **Action**: Delete `src/wire_test.ts`. 
- **Validation**: Ensure `npm run build` succeeds and the project is still correctly invokable via `nix develop --command npx tsx src/dispatch.ts`. Verify that `src/index.ts` and Express dependencies remain intact.

### Increment 2: Expand Core Utility Tests with Meaningful Validation
Implement unit tests for core utilities using a strategy that avoids over-mocking. Rather than asserting on mock function calls (which tightly couples tests to implementation details), tests should validate actual behavior via HTTP interceptors or real file system interactions.
- **Action**: Create `src/utils/gemini.test.ts`. Use an HTTP interceptor (e.g., `nock` or Node's native fetch mocking) to stub out the Gemini API endpoints. This ensures the actual HTTP request shape (headers, payload) generated by the service is validated.
- **Action**: Create `src/utils/github.test.ts`. Validate GitHub API operations (data retrieval, issue reading, comment posting) using realistic HTTP response fixtures and intercepted requests, ensuring the service correctly processes authentic JSON shapes.
- **Action**: Create `src/utils/prompt_files.test.ts`. Point the utility at a temporary directory containing actual markdown templates to verify that file loading and variable substitution work correctly against the real file system APIs.
- **Action**: Create `src/utils/trace.test.ts`. Validate the `AsyncLocalStorage` behavior by executing actual asynchronous workflows and asserting the trace context is preserved throughout the call stack.
- **Validation**: Ensure `npm test` passes on all newly added test files.

### Increment 3: Expand Script Tests
- **Action**: Create `src/scripts/run_task_persona.test.ts` to test CLI argument parsing (`parseArgs`) and behavior configuration. Rather than mocking the runner, capture stdout/stderr or use an integration-style test to ensure the script wires dependencies correctly.
- **Validation**: Ensure `npm test` successfully executes the script's tests.
